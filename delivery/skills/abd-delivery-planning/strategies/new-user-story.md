# Strategy: New User Story

**When to use:** A small change on an existing solution — new feature, enhancement, or capability within the current architecture. Map, slices, and domain model already exist.

**Typical scope:** 1–3 stories in one backlog increment (or a micro-increment).


| Step | Stage | Intent | Scope | Checkpoint policy | Rationale |
| ---- | ----- | ------ | ----- | ----------------- | --------- |
| 1 | Exploration → Specification → Engineering | **Deliver each story end-to-end:** UL/AC → CRC/spec scenarios → acceptance tests → implementation | New story or stories, one at a time | Per-story within each role; cross-role after each story completes the chain | Small scope, full pipeline; each story earns its quality bar |


**Key constraints:**

- Read the story graph and corrections log first; stay consistent with domain language.
- Surface prior corrections to anyone touching the story.
- If the map itself must be restructured, switch strategy (*New Thin Slice* / *New Initiative*).
