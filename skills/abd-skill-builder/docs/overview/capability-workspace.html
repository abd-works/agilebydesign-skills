Capability 1 of 8

Workspace & configuration

Every script and instruction in your skill knows where to conduct its work.

Problem
Skill writes in the wrong area; reads and writes wrong files. Doesn't pick up your last work.
Solution
A single skill-config.json declares the active workspace so every script and instruction in your skill knows where to conduct its work.

What "workspace" means

The pointer chooses the project root for everything the skill produces or updates outside its own package: plans, pipeline checklists, generated docs, deliverables, and other engagement artifacts. Those paths resolve under that tree, not under the skill install. The pointer tells every script and instruction which tree is live. 

        Skill package (install)                   Workspace (project tree)
        read/write: the skill's own files         read/write: engagement artifacts
┌─────────────────────────────────────┐     ┌─────────────────────────────────────┐
│ Installed package — SKILL.md,       │     │ Project tree — plans, pipeline      │
│ scripts/, content/, rules/, tests…  │ ──► │ checklists, generated docs,         │
│                                     │     │ deliverables, runtime state, etc.   │
└─────────────────────────────────────┘     └─────────────────────────────────────┘
                      skill-config.json → workspace.active_skill_workspace

Key fields in skill-config.json

  • workspace.active_skill_workspace — absolute path to the current project root
  • workspace.known_skill_workspaces — optional list of other valid roots for switching
  • workspace.context_paths — extra read-only paths (reference docs, shared libraries)

Helper script

scripts/base/set_workspace.py — prints the active workspace or writes a new pointer to skill-config.json. Run it at session start to confirm the skill is aimed at the right tree.
