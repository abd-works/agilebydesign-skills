Rules & scanners

Separate rule files injected into prompts; scanners act as x-ray vision for AI slop.

Problem
AI stubbornly makes the same mistakes despite the best prompting. Guidance gets scattered across the skill over time. AI constantly requires close supervision to get to quality.
Solution
Decompose rules into separate files, guide AI by injecting rules into prompts, enforce rules mechanically through scanners — scripts that check for syntactic smells and act as x-ray vision for AI slop.

Two-layer architecture

┌──────────────┐   read at prompt time   ┌──────────────────┐
│  rules/*.md  │ ─────────────────────►  │  AI follows      │
│  (prose)     │                         │  rule text       │
└──────────────┘                         └──────────────────┘

┌──────────────┐   run after merge       ┌──────────────────┐
│ scanners.json│ ─────────────────────►  │  Automated check │
│  (bindings)  │                         │  flags violation │
└──────────────┘                         └──────────────────┘

How scanners bind to rules

rules/scanners.json lists entries like: 

{
  "scanners": [
    {
      "rule": "rules/naming-conventions.md",
      "script": "scripts/scanners/check_naming.py",
      "on": ["build", "ci"]
    }
  ]
}
  • rule — the prose rule the scanner enforces.
  • script — the Python script that checks the output.
  • on — when to run: build (local), ci, or both.

What happens when a scanner fires

A violation is logged with the rule it broke and the location in the output. Feed findings into the correction approach — a problem log and improvement backlog to prevent recurrence.
