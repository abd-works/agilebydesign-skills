## Overview

Teams and organisations routinely carry unverified assumptions about customers, value, feasibility, and economics into delivery. This skill surfaces those assumptions explicitly, rewrites them as falsifiable hypotheses, and prioritises them into a validation backlog so the cheapest, highest-risk items are tested first.

The output is not a research programme — it is a small, accountable set of experiments, each with an owner, a named method, and an explicit failure condition. Work moves through **Plan → Validate → Learn** before the team commits to a full build.

Inputs can come from anywhere: an opportunity canvas, impact map, business case, cost-of-delay estimate, journey map, or workshop notes. The skill mines whatever is provided, converts assumptions to hypotheses, and structures each one so the team knows what they are testing, how, and what they will do with the result.
